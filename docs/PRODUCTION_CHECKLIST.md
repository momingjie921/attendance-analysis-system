# 生产上线检查清单

本清单用于判断系统是否可以从灰度试运行进入生产使用。当前项目适合中小企业进行最小可用上线，但不建议在未完成以下事项前直接替代企业唯一考勤系统。

## 1. 配置安全

- `.env` 不提交到 GitHub。
- `SECRET_KEY` 使用随机强密钥。
- `DATABASE_URI` 使用独立数据库账号，不使用 root。
- `SMTP_PASS` 使用邮箱授权码或密钥管理系统，不写入代码仓库。
- `FLASK_DEBUG=False`。
- 生产环境启用 HTTPS。
- `SESSION_COOKIE_SECURE=True`。
- `API_CSRF_PROTECT=True`。
- 登录表单和管理类 API 均保留 CSRF 校验。

## 2. 账号与权限

- 删除或重置所有演示账号密码。
- SQL 种子数据仅用于测试，不能作为正式员工主数据来源。
- 管理员账号数量最小化。
- 部门经理只能查看和处理本部门数据。
- 员工只能查看本人考勤、异常和请假记录。
- 新增员工默认密码必须满足强度要求，并要求首次登录后修改。
- 离职员工账号及时停用。

## 3. 数据库

- 数据库字符集使用 `utf8mb4`。
- MySQL 8.0 推荐排序规则 `utf8mb4_0900_ai_ci`。
- MySQL 5.7 或兼容场景使用 `utf8mb4_unicode_ci`。
- 定期备份数据库，并做过恢复演练。
- 导入大批量考勤数据前先备份。
- 关键表保留必要索引，避免重复索引。
- 请假日期、请假天数、状态字段存在基础约束。

## 4. 数据质量

- 明确迟到、早退、缺勤、请假、节假日和调休规则。
- 导入模板字段固定，并保留样例模板。
- 导入后抽样核对原始打卡数据与系统计算结果。
- 异常结论必须经过 HR 或管理员复核。
- 规则变更后重新验证历史样本。

## 5. 运行监控

- 启用应用日志和错误日志。
- 关键管理操作写入审计日志。
- 导入失败、备份失败和接口异常有告警渠道。
- 定期检查磁盘容量、数据库连接数和慢查询。
- 保留至少 30 到 90 天的运行日志，具体周期按公司制度确定。

## 6. 灰度验收

正式上线前建议完成：

- 至少 2 周灰度试运行。
- 覆盖 1 到 2 个真实部门。
- 和现有考勤流程并行核对。
- 统计结果误差原因可解释、可修复。
- 管理员、经理、员工三类角色均完成验收。
- 出现异常时有人工兜底流程。

## 7. 当前结论

完成本仓库已有最小上线改造后，系统可以作为中小企业考勤数据分析工具进行灰度使用。若要承担正式生产职责，还需要结合企业制度完成权限验收、备份恢复、异常复核、数据导入规范和运维监控。
